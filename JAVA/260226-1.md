# ☕ 자바의 문자 처리 매커니즘과 언어적 특징 분석

자바는 내부적으로 모든 문자를 유니코드로 처리하며, 이는 C언어와 비교했을 때 메모리 관리와 네트워크 전송 방식에서 큰 차이를 보입니다.

---

## 1. 자바의 문자 처리 (내부 메모리 vs 전송)

자바는 문자를 처리할 때 메모리 내부와 외부 출력 시의 인코딩 방식을 다르게 적용합니다.

[Image of Unicode UTF-16 vs UTF-8 encoding comparison for Korean characters]

### 📍 자바 내부 메모리 (UTF-16)
* 자바의 `char`와 `String` 연산(charAt, length 등)은 **UTF-16 코드 유닛** 기반입니다.
* 한글('가', '한' 등)은 BMP 영역에 속하여 **2바이트(1개 유닛)**로 처리됩니다.

### 📍 외부 저장 및 전송 (바이트 스트림)
* 파일 저장이나 네트워크 전송 시에는 `byte[]`로 변환(인코딩)해야 합니다.
* **인코딩 방식에 따른 한글 1글자의 크기:**
    * **UTF-8:** 3바이트 (현재 가장 표준적인 방식)
    * **EUC-KR / UTF-16:** 2바이트
* **예시:** `"한".getBytes("UTF-8").length` 결과값은 **3**이 됩니다.

## 2. 언어별 문자열 저장 및 형변환 비교

### 1) 문자열 저장 방식
* **C:** `char *a = "사과";` (메모리 주소를 가리키는 포인터 사용)
* **Java:** `String a = "사과";` (클래스 객체 사용)

### 2) 형변환 (C 동적 할당 예시)
* **C:** `int *a = (int *)malloc(sizeof(int));` ➔ `void *` 타입을 `int *`로 명시적 형변환.
* **Java:** 자바도 객체 간 상속 관계에서 명시적/암시적 형변환을 사용합니다.

---

## 3. 라이브러리 호출 및 컴파일 특징

### 📍 include vs import
* **C / C++:** `#include`를 사용하여 헤더 파일을 포함합니다.
* **Java:** `import`를 사용하여 패키지 내의 클래스를 가져옵니다. (C++ 최신 문법에서도 `import` 지원 시작)

### 📍 메서드 위치의 자유로움
* **C:** 호출되는 함수가 호출하는 곳보다 위에 있거나, 프로토타입 선언이 반드시 필요합니다.
* **Java:** 메서드 호출 후 아래에 클래스나 메서드 정의가 있어도 상관없습니다. 컴파일러가 소스 전체를 한 번에 분석하기 때문입니다.

---

## 4. Scanner 클래스와 패키지 구조

자바에서 입력을 담당하는 `Scanner`는 특정 패키지에 소속되어 있습니다.

[Image of Java package hierarchy structure showing java.util.Scanner]

* **구조 분석: `java.util.Scanner`**
    * `java`: 최상위 패키지
    * `util`: 하위 패키지
    * `Scanner`: 클래스 이름
* **주의사항:** `Scanner` 사용 후에는 반드시 `객체.close()`를 호출하여 스트림을 닫아줘야 리소스 누수(Warning)를 방지할 수 있습니다.

## 5. C언어 분할 컴파일의 자바식 변환 예시

C언어의 파일 분할 구조를 자바의 클래스 구조로 매핑하면 다음과 같습니다.

| C 파일 및 함수 | Java 클래스 및 메서드 |
| :--- | :--- |
| `main.c` / `main()` | `Main.java` / `main` 메서드 |
| `test_func1.c` / `test_func1()` | `TestFunc1.java` / `testFunc1` 메서드 |
| `test_func2.c` / `test_func2()` | `TestFunc2.java` / `testFunc2` 메서드 |

2. `a = a + 1;`
3. `int a = 1; a = a + 1;` (초기화 포함)
